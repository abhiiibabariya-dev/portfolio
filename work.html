Case 01

Incident Response · Cloud Forensics · APT Investigation

State-Sponsored APT Alert Investigation

Multi-source forensic investigation of a suspected North Korea–linked APT communication reaching a production cloud workload.

Azure Defender for CloudAzure Log Analytics KQLSentinelOne EDRMicrosoft Defender for Endpoint Windows Event LogsPowerShell

The Challenge

A cloud-native detection platform flagged a medium-severity alert: a monitored production endpoint had made a DNS query to a domain attributed to a known nation-state APT. By the time the investigation started, volatile endpoint telemetry — DNS cache, short-retention Windows logs — had already rotated. The forensic picture had to be rebuilt from durable cloud telemetry and the residual artefacts still on the host.

The Approach

I ran a structured five-phase investigation designed to cover both endpoint and cloud sources so no single telemetry gap could mask a compromise:

  • Phase 1 — Live endpoint forensics. Active connections, running processes, scheduled tasks, registry run keys, filesystem staging locations, and browser history across every user profile on the host.
  • Phase 2 — Defender operational log review at scale. Parsed ~17,000 events from the Windows Defender log across a 60-day window looking for detections, ASR triggers, tamper-protection anomalies, and signature-update health.
  • Phase 3 — Cloud SIEM investigation. Authored targeted KQL queries against ~900,000 forwarded Windows events — a time-windowed process-creation sweep around the alert, plus full-dataset searches for the IOC across a 30-day range.
  • Phase 4 — EDR Deep Visibility search across the incident window to check for any DNS or connection artefacts the endpoint had captured.
  • Phase 5 — Alert-graph reconstruction via cloud Resource Graph queries to try to recover the original alert payload and its embedded evidence.

Key Findings

  • Zero malware detections across three independent security products running on the host.
  • No persistence mechanisms, no lateral-movement indicators, no data-exfiltration evidence.
  • Every process observed within the incident window ran under the SYSTEM context — no user-initiated activity to correlate a phishing click against.
  • Alert correctly classified as True Positive · No Compromise — the detection was valid, the impact was zero.

Root Cause Analysis

Documented a confidence-graded hypothesis (most / less / least likely) rooted in the evidence actually available. The definitive source process could not be attributed because of telemetry gaps that existed in the environment before the alert fired — and that limitation was itself documented as an operational finding rather than glossed over.

Remediation Roadmap Delivered

Tiered recommendations across three horizons — the point wasn't just to close a ticket, but to fix the systemic issues that made this investigation harder than it should have been.

  • Immediate (48 hrs): block the IOC at firewall and DNS layers, retrieve the full alert payload from the cloud console, and pivot to a hunting portal capable of answering the process-attribution question definitively.
  • Short-term (30 days): Sysmon deployment for Event ID 3 & 22 (network + DNS with process attribution — the single control that would have resolved this in minutes), Windows event-log retention uplift, DNS-client operational log retention, and enablement of EDR DNS event collection.
  • Long-term (90 days): cloud DNS Analytics and NSG Flow Logs, centralised SIEM forwarding with 90-day retention, SOAR playbooks for malicious-domain alerts to close the response-time gap, and a formal SLA for medium-severity cloud alerts.

Impact

The alert closed with high-confidence disposition, and the environment walked away with a maturity-uplift roadmap. A single alert became a systemic capability improvement for the SOC's cloud visibility.

Case 02

SIEM · SOAR · Automation

SOC Automation Lab

A production-grade orchestration stack that cut manual triage time by 80%.

WazuhTheHiveShuffle SOAR VirusTotal APIAbuseIPDBMITRE ATT&CK

The Challenge

SOC analysts routinely drown in noise. A mid-sized environment can push thousands of alerts per day, and the majority of triage time is spent on repetitive enrichment tasks — checking IPs against threat-intel sources, pivoting between tools, and copy-pasting IOCs into ticketing systems. That mechanical work squeezes out time for the actual investigation.

The Approach

I architected an end-to-end automation lab integrating three complementary tools: Wazuh as the SIEM for centralized log collection and correlation, TheHive as the case-management layer for structured investigation, and Shuffle SOAR as the orchestration engine that stitches everything together.

Detection rules in Wazuh trigger enriched alerts that Shuffle picks up. Playbooks fan out to VirusTotal and AbuseIPDB for reputation lookups, extract IOCs, calculate a severity score, then create a fully-populated case in TheHive — evidence attached, MITRE tactics tagged, containment task queued.

Playbooks Built

  • Brute-force authentication — detects repeated failed logins, correlates with successful auth from the same source, auto-creates a case with attacker IP, target account, and geolocation.
  • Malware execution — Wazuh flags suspicious process creation, playbook queries VirusTotal for hash, isolates the endpoint if score exceeds threshold.
  • File Integrity Monitoring — unauthorized changes to critical files trigger evidence collection, snapshot of the process tree, and analyst notification.

The Outcome

  • 80% reduction in manual triage time per alert.
  • Fully populated cases in TheHive land with enrichment already done — analyst opens a case with context, not a blank ticket.
  • MTTD & MTTR dashboards established for operational visibility.
  • Reusable playbook library that maps 1-to-1 with MITRE ATT&CK tactics.

Case 03

Detection Engineering

CyberGuard — Threat Detection System

Custom detection signatures validated against red team scenarios.

Detection RulesBehavioral Analytics VirusTotalANY.RUNYARARegex

The Challenge

Off-the-shelf detection rules catch the loud, obvious attacks — but modern adversaries live off the land. PowerShell obfuscation, WMI abuse, scheduled tasks for persistence, and credential dumping via legitimate binaries are all invisible to signature-only systems.

The Approach

I combined regex-based lexical detection with behavioral analytics to catch what static rules miss. The detection library targets:

  • Brute-force authentication — threshold-based failed login patterns with time-window correlation.
  • Malware execution indicators — suspicious process spawning, unsigned binary execution, obfuscated command lines.
  • Unauthorized access — privilege escalation, lateral movement, off-hours access anomalies.

Triage Pipeline

Each detection funnels through a triage workflow. Suspicious samples get detonated in VirusTotal and ANY.RUN — I observe runtime behavior, capture network communication, identify persistence mechanisms, and extract fresh IOCs that feed back into the detection ruleset. It's a closed loop.

Signatures Authored

  • PowerShell obfuscation (Base64, char-code, XOR encoded payloads).
  • WMI-based lateral movement and persistence.
  • Scheduled-task creation with encoded commands.
  • Credential dumping via LSASS access, comsvcs.dll abuse, and legitimate-binary misuse (LOLBins).

The Outcome

All signatures were validated against red team engagement scenarios and tuned to minimize false positives. Detection library is portable across SIEM platforms via Sigma-style rule mapping.

Case 04

Digital Forensics · DFIR

Mobile & Cloud Forensics Investigation

Legal-grade forensic examinations across mobile devices and cloud services.

AutopsyFTK ImagerADB Cloud ForensicsChain of Custody

The Challenge

Modern investigations rarely involve a single device. Suspects sync data across a phone, a laptop, and half a dozen cloud services — Google Drive, iCloud, Dropbox, OneDrive. A defensible forensic examination has to preserve integrity, reconstruct timelines across all of those sources, and produce reports that hold up in legal proceedings.

Mobile Acquisition

Performed bit-by-bit forensic imaging and logical extraction using industry-standard tools:

  • Autopsy for disk image analysis, timeline construction, and keyword search.
  • FTK Imager for forensic acquisition with hash-verified chain-of-custody.
  • ADB (Android Debug Bridge) for live device data extraction and logical dumps.

Artifacts recovered included SMS/MMS, call logs, application databases, GPS location history, and deleted files carved from unallocated space.

Cloud Sync Analysis

Extended the investigation into cloud-sync footprints — recovering authentication tokens, file metadata, revision histories, and deleted cloud items. Reconstructed user activity timelines across services to establish attribution.

The Outcome

  • Complete forensic methodology documented with hash-verified evidence exhibits.
  • Chain-of-custody records maintained end-to-end.
  • Investigation reports formatted for legal proceedings with expert findings and timeline analysis.

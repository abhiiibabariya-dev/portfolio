State-Sponsored APT Alert Investigation

Suspected North Korea–linked APT beaconing detected against a production cloud workload — closed True Positive · No Compromise.

Client
Enterprise · Cloud Workload
Duration
72 hours
Role
Lead Investigator
Verdict
True Positive · No Compromise
Azure Defender for Cloud Azure Log Analytics KQL SentinelOne MDE Threat Intel

Overview

Azure Defender for Cloud flagged an outbound connection from a production VM to an IP reputationally linked to a North Korea–nexus APT cluster. I led a five-phase forensic investigation across live endpoint data, ~17K Defender log events, and ~900K events queried via KQL in Azure Log Analytics — reconciling network telemetry, EDR process trees, and threat-intel enrichment to reach a defensible verdict within 72 hours.

Problem

  • High-severity alert: outbound TLS to an IP historically associated with Lazarus-cluster infrastructure.
  • No corroborating EDR detection at initial triage — required deep multi-source correlation.
  • Regulated workload — every action needed chain-of-custody grade documentation.

Phases

01
Scoping & Preservation
  • Isolated the affected VM at the NSG level while preserving live memory and process state.
  • Snapshotted OS disk and captured volatile artifacts (netstat, tasklist, prefetch, event logs).
  • Established chain-of-custody documentation and evidence hashing (SHA-256).
02
Endpoint Forensics (MDE + SentinelOne)
  • Reviewed MDE Advanced Hunting timeline for the 48h window preceding the alert.
  • Reconstructed process trees around the outbound TLS event — parent lineage traced to a signed vendor service.
  • Cross-checked SentinelOne DeepVisibility for the same PID / connection tuple.
03
Log Analytics — ~900K events via KQL
  • Queried SecurityEvent, DeviceNetworkEvents, and SigninLogs across a two-week window.
  • Correlated 4624 / 4672 / 4688 with DNS + firewall telemetry to look for lateral movement precursors.
  • Wrote KQL joins to hunt for beacon regularity (jitter windows, sub-second reconnects).
04
Threat-Intel Enrichment
  • Enriched the destination IP via VirusTotal, AbuseIPDB, and internal TI feeds.
  • Reviewed passive DNS and historical WHOIS — identified the IP had been repurposed via a legitimate CDN.
  • Mapped observed TTPs against MITRE ATT&CK T1071.001 & T1573 to test the APT hypothesis.
05
Verdict & Remediation Roadmap
  • Confirmed the traffic originated from a legitimate signed telemetry client using a shared CDN edge.
  • Closed as True Positive (rule fired correctly) · No Compromise (no adversary activity).
  • Delivered tiered remediation: signature tuning, egress allow-lists, and hunting queries for future coverage.

Outcomes

~900K

0
Adversary artifacts found

Learnings

  • Reputation alone is a starting point, not a verdict — signed telemetry on shared CDN edges can trigger APT-labelled indicators.
  • Beacon-jitter KQL patterns are worth pre-building as saved hunts; they collapsed hours of triage into a single query.
  • A tiered remediation write-up (detect / prevent / hunt) is more actionable for stakeholders than a single fix.
02 · SIEM · SOAR · Automation

SOC Automation Lab — Wazuh + TheHive + Shuffle

A production-grade SOC automation stack that reduced manual triage time by 80% through playbook-driven response.

Client
Personal Lab / Portfolio
Duration
6 weeks
Role
Architect · Detection · Automation
Verdict
Manual triage ↓ 80%
Wazuh TheHive Cortex Shuffle SOAR VirusTotal AbuseIPDB MITRE ATT&CK

Overview

Designed and deployed a fully integrated SOC automation environment: Wazuh for log ingestion and detection, TheHive for case management, and Shuffle for SOAR orchestration. The lab ingests real endpoint & network telemetry, enriches indicators via VirusTotal and AbuseIPDB, and auto-creates cases with pre-populated observables and playbook-driven response steps.

Problem

  • Manual alert triage was consuming disproportionate analyst time on repetitive enrichment steps.
  • Alerts, cases, and enrichment lived in disconnected tools with no shared context.
  • No consistent way to measure playbook effectiveness or false-positive rate over time.

Phases

01
Architecture & Ingestion
  • Deployed Wazuh manager + indexer + dashboards on a hardened Ubuntu host.
  • Onboarded Windows, Linux, and Sysmon agents; ingested Suricata NIDS events via rsyslog.
  • Structured indexes for hot/warm tiers and configured retention aligned with case-review needs.
02
Detection Engineering
  • Authored custom Wazuh rules for PowerShell obfuscation, WMI abuse, and scheduled-task persistence.
  • Mapped every rule to MITRE ATT&CK sub-technique for coverage tracking.
  • Tuned noisy defaults; whitelisted expected admin activity to slash false positives.
03
TheHive + Cortex Case Management
  • Configured TheHive with a Wazuh webhook — every high-severity alert auto-generates a case.
  • Connected Cortex analyzers (VirusTotal, AbuseIPDB, MISP) to enrich observables on ingest.
  • Built case templates with pre-populated tasks aligned to the PICERL lifecycle.
04
Shuffle SOAR Playbooks
  • Built playbooks: phishing triage, brute-force response, malware execution, and IOC sweep.
  • Playbooks call VirusTotal, geo-locate source IPs, post Slack notifications, and update TheHive.
  • Added human-in-the-loop gates for containment actions (e.g., isolate host / reset password).
05
Validation & Metrics
  • Simulated Atomic Red Team techniques to validate rule + playbook coverage.
  • Measured mean-time-to-triage before & after — 80% reduction on repeatable playbook paths.
  • Documented runbooks and onboarding notes for future analysts.

Outcomes

Learnings

  • SOAR value compounds — the second playbook is 3× faster to build than the first because primitives are shared.
  • Case templates aligned to PICERL keep analyst investigations consistent without slowing them down.
  • Enrichment on ingest (not on demand) is what actually saves the time; analysts open cases already 70% resolved.
03 · Detection Engineering

CyberGuard — Threat Detection System

Behavioral + regex detection logic hunting PowerShell obfuscation, WMI abuse, and credential-dumping — validated against red-team scenarios.

Client
Personal Research
Duration
4 weeks
Role
Detection Engineer
Verdict
Red-team validated
Detection Rules Behavioral Analytics YARA VirusTotal ANY.RUN Sysmon

Overview

Engineered a compact but production-minded detection library targeting living-off-the-land techniques that traditional signature engines miss. Rules combine regex over command-line telemetry with behavioral indicators (parent-child anomalies, uncommon Sysmon event chains) to catch PowerShell obfuscation, WMI persistence, scheduled-task abuse, and LSASS credential access.

Problem

  • Adversaries increasingly abuse signed system binaries — signature-only detection is blind to these paths.
  • Analyst inbox noise from off-the-shelf rules erodes trust in the SIEM.
  • Needed a testable, reproducible way to prove each rule works against a real technique.

Phases

01
Technique Selection
  • Selected 8 high-signal ATT&CK sub-techniques common across recent incident reports.
  • For each, listed the minimum telemetry (Sysmon events, Windows Event IDs, EDR fields) required to detect it.
  • Prioritized techniques with high dwell-time impact and low native coverage.
02
Rule Authoring
  • Built regex signatures for PowerShell obfuscation: base64 blocks, invocation via -EncodedCommand, char-code obfuscation, and IEX chains.
  • Wrote behavioral rules for WMI abuse — wmic.exe / WmiPrvSE.exe spawning non-standard children.
  • Added detections for scheduled-task persistence (Event ID 4698 with unusual authors) and LSASS access from non-signed binaries.
03
Enrichment & Context
  • On rule hit, auto-lookup file hashes against VirusTotal and detonate suspicious samples in ANY.RUN.
  • Attach parent-process lineage and command-line to the alert body so triage doesn't require a pivot.
  • Add a static ATT&CK tag on every rule for coverage reporting.
04
Red-Team Validation
  • Ran Atomic Red Team, PowerShell Empire, and manual WMI persistence tests in an isolated lab.
  • Tuned rules against false-positive baselines drawn from a 7-day clean telemetry sample.
  • Documented every rule with the exact test it passes, so regressions are catchable.

Outcomes

8
ATT&CK techniques covered
0
Missed red-team detections
< 2%
Baseline false-positive rate
100%
Rules with a validating test

Learnings

  • Every rule should ship with its own reproducible test — otherwise regressions silently rot coverage.
  • Behavioral joins (parent + child + args) beat any single-field regex on modern living-off-the-land techniques.
  • Enrichment context inside the alert body is the single biggest ergonomic win for downstream analysts.
04 · Digital Forensics · DFIR

Mobile & Cloud Forensics Investigation

Bit-by-bit imaging + cloud sync analysis producing legal-grade reports with full chain-of-custody.

Client
Academic Case Study · NFSU
Duration
3 weeks
Role
Lead Examiner
Verdict
Legal-grade chain of custody
Autopsy FTK Imager ADB Google Drive iCloud Dropbox OneDrive

Overview

End-to-end mobile + cloud forensic examination combining bit-by-bit device imaging with cloud artifact analysis across four providers. Deliverables included a legal-grade report suitable for evidentiary use with complete chain-of-custody documentation, hash verification at every stage, and reproducible extraction procedures.

Problem

  • Modern investigations require both device-resident and cloud-resident artifacts — most examiners cover only one.
  • Chain-of-custody breaks are the single most common reason evidence is dismissed.
  • Deleted artifact recovery on modern Android/iOS requires careful acquisition to avoid overwriting.

Phases

01
Acquisition
  • Powered device isolated via faraday bag; performed logical + physical extraction with ADB and vendor tools.
  • Created bit-by-bit forensic images with FTK Imager; verified SHA-256 hashes before and after transfer.
  • Documented every action with timestamps and examiner signature.
02
Device Artifact Analysis (Autopsy)
  • Parsed SMS, MMS, call logs, and contact stores from SQLite databases.
  • Recovered deleted photos and documents from unallocated space and SQLite free-lists.
  • Reconstructed GPS history from location caches and correlated with photo EXIF metadata.
03
Cloud Sync Analysis
  • Analyzed Google Drive, iCloud, Dropbox, and OneDrive sync artifacts on the device.
  • Correlated cloud-side timestamps with device-side sync markers to build a unified activity timeline.
  • Documented file lineage — created on device, synced to cloud, restored from cloud, or downloaded fresh.
04
Reporting & Chain of Custody
  • Produced a legal-grade report with executive summary, methodology, artifacts, and appendices.
  • Bundled a chain-of-custody log covering seizure → acquisition → analysis → storage.
  • Delivered a reproducibility appendix listing every tool version and exact command used.

Outcomes

100%
Chain-of-custody preserved
4
Cloud providers covered
SHA-256
Hash-verified at every stage
Legal
Grade of final report

Learnings

  • Hash on acquisition, hash on transfer, hash on analysis — three points is the minimum for defensible evidence.
  • Cloud sync markers are a goldmine — they let you time-anchor artifacts you'd otherwise treat as ambiguous.
  • The reproducibility appendix (tool versions + exact commands) is what separates a good report from an admissible one.
